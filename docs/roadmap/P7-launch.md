# P7: 公開準備（ローンチ）

> v1 を「人に使ってもらえる状態」にする最終フェーズ。品質・性能・導線・運用を仕上げて公開する。

- **依存**: P1〜P6
- **領域**: `infra` / `webfront` / `mobile` / `uiux`
- **完了でアンロックされるもの**: v1 公開

## 目的

- コアフロー（集める→AI補完→クイズ/カード→定着）の品質を保証する。
- 性能・コスト（AI/ベクトル/R2/Redis）を許容範囲に収める。
- ランディングページとオンボーディングを整え、新規ユーザーが迷わない。
- 本番デプロイ・監視・障害対応の運用を用意する。

## スコープ

### やる
- E2E/手動QA（主要ユーザーフローのシナリオテスト）。
- 性能・コスト点検（AI呼び出し削減のキャッシュ効果、ベクトル検索レイテンシ、画像配信）。
- ランディングページ（`apps/web` の `index.tsx`）とオンボーディング/プリセット初期導線。
- 本番デプロイ（Cloudflare Workers / Turso / R2 / Redis の本番設定）と環境変数整理。
- エラーモニタリング・ログ・基本的なアラート。
- 法務/規約（プライバシーポリシー・利用規約）、課金の本番化。
- モバイルのストア提出物（アイコン/スクショ/メタ）と審査対応（提出する場合）。

### やらない
- v1 後の追加機能（「一年前の今日」通知、word cascade の高度演出、ML最適化など）。

## データモデル差分

- なし（仕上げフェーズ）。

## API 仕様

- 新規なし。必要に応じてヘルスチェック `GET /health` とレート制限の本番調整。

## 主要UI / 画面

- ランディングページ。
- オンボーディング（初回プリセット選択〜最初の言葉登録までの導線）。
- エラー/空状態/ローディングの仕上げ。

## タスク（Issue 化）

1. 主要フローの E2E/手動QAシナリオ作成と実行
2. 性能・コスト点検（キャッシュヒット率/ベクトル検索/画像配信）と改善
3. ランディングページ仕上げ
4. オンボーディング（初回プリセット〜初回登録）導線
5. 本番デプロイ構成と環境変数・シークレット整理
6. エラーモニタリング/ログ/アラート導入
7. プライバシーポリシー・利用規約・課金本番化
8. （モバイル提出する場合）ストア提出物と審査対応

## 完了条件

- 主要フローが本番相当環境で安定動作する。
- 性能・コストが許容範囲に収まることを確認済み。
- ランディング/オンボーディングが整い、新規ユーザーが登録〜学習まで到達できる。
- 本番デプロイ・監視・規約が整い、**v1 公開**できる状態。
